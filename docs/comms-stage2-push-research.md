# Push-Based Inter-Agent Communication for Claude Code Agents (with File-Queue Fallback)

## TL;DR
- **The cleanest push mechanism that exists today is a "blocked-Stop-hook + local broker" pattern: each agent runs a Stop hook that, instead of letting the turn end, long-polls a local message bus (Redis/NATS/file) for up to ~30s and re-injects any message as `additionalContext`, so the agent "stays awake" and reacts in near-real-time — and falls back to your existing file queue the moment it does stop.** There is no native "inject into an idle/mid-session Claude Code agent" API; true server-initiated push into Claude Code is a confirmed gap (anthropics/claude-code issue #36665).
- **On Windows 11 specifically, the tmux `send-keys` injection that works on Mac/Linux is unavailable, and the VSCode extension chat panel cannot be injected into at all (issue #27873, closed unimplemented).** Your realistic Windows options are: (a) the Stop-hook self-wake loop above, (b) `psmux` (a native ConPTY tmux clone) to recover true `send-keys`, or (c) AutoHotkey/SendKeys with focus theft. The Ink TUI ignores programmatic `\r`/`\n`, so naive "pipe text + Enter" does NOT submit (issue #15553).
- **Recommended architecture for a single dev with 4–5 dynamic agents: run one lightweight local broker (Redis or NATS, single binary) as the message bus, give each agent (1) a Stop/UserPromptSubmit/SessionStart hook that drains its inbox and (2) a tiny sidecar that maintains presence via heartbeat and writes to your existing file queue when the target is offline.** Adopt Anthropic's official **Agent Teams** (research preview, `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`) if and only if you can move your CLI agents to Mac/Linux or psmux — its automatic message delivery solves push natively but does not work in the VSCode panel or Windows Terminal split panes.

## Key Findings

### 1. There is no native push into a running/idle Claude Code agent — this is the core constraint
Claude Code operates in a request/response loop. When a turn ends (`decision: stop`), the session is effectively idle and is **not running an event loop that consumes inbound messages**. Multiple primary sources confirm this:
- **MCP server→client push is received by the CLI but not surfaced to the model** unless the user types something. This is documented in the open feature request **anthropics/claude-code #36665** ("feat: MCP server push notifications"), filed by the team behind the `synapt` memory system, who state: *"The lack of server push is the biggest remaining coordination gap — agents go 'dark' when idle because they have no way to receive notifications without a tool call."* The MCP spec defines `notifications/message`, and FastMCP exposes `ctx.session.send_progress_notification(...)`, but **Claude Code does not pipe these back into the model's prompt**.
- **No injection API exists.** Two open feature requests — **#24947** (`claude inject <session_id> --text "..."`) and **#27441** (inter-agent message injection / a file-watch `InboxMessage` hook event) — explicitly ask Anthropic to add this and confirm it does not exist. #27441's author notes their `claude --print` + git-watch daemon approach "works but loses interactive session continuity," and that today "true agent-to-agent communication requires the human to be the relay."
- **The Ink TUI ignores programmatic Enter.** Issue **#15553** documents that a physical Enter triggers `onSubmit`, but a programmatic `\r`/`\n` is treated as a newline and does NOT submit. Nine injection approaches failed; the only working terminal workaround is **bracketed-paste mode**: `terminal.sendText("\x1b[200~" + prompt + "\x1b[201~", false)` then a short wait, then send an empty line to submit.

**Dead ends to avoid:** (a) expecting an MCP server to "notify" an idle agent into action; (b) piping text to a running CLI's stdin expecting it to submit; (c) any programmatic input to the VSCode extension chat *panel*.

### 2. What CAN deliver a message into an agent — three real mechanisms

**(A) Hook-based self-wake / inbox-drain (most portable, no keystroke injection).** Claude Code's hook system is the single most reliable lever because certain hooks inject text directly into the model's context:
- **`SessionStart` and `UserPromptSubmit`**: plain stdout is injected as conversation context. `SessionStart` can also return `watchPaths` (to arm `FileChanged`), `additionalContext`, and even `initialUserMessage`.
- **`Stop` / `SubagentStop`**: returning `{"decision":"block","reason":"..."}` (or exit 2) **refuses to let the turn end and feeds the reason back to the model**, optionally with `additionalContext`. This is the key to "staying awake."
- **`FileChanged`**: fires on native OS filesystem events (FSEvents/inotify) when a watched file (matched by basename) changes; it is **command-only** and is observability-oriented — it cannot inject `additionalContext` the way SessionStart/UserPromptSubmit can, so it is best used to *trigger a sidecar*, not to deliver the message itself.
- **`TeammateIdle` / `TaskCompleted`**: experimental agent-team hooks; `TeammateIdle` fires when a teammate goes idle and can inject "pick up the next task" context.

The production pattern (DEV "agent-room" post): a **Stop hook that long-polls a local bus for ~30s** (e.g., Redis `BRPOPLPUSH`-style blocking wait). If a message arrives, it returns `decision: block` with the message as the reason → the agent processes it that turn. If nothing arrives in 30s, it returns exit 0 and the agent stops cleanly (and your file queue is the fallback). Must honor `stop_hook_active` and cap wake-ups (`MAX_BLOCKS_PER_CYCLE`) to avoid infinite loops — issue **#55754** documents a Stop-hook loop that burned a full session quota (~50 min) when it never returned `ok`.

**(B) Terminal keystroke injection (true push, OS-dependent).**
- On **Mac/Linux with tmux/iTerm2**: `tmux send-keys -t <pane> -l "msg"` + `Enter` "arrive reliably with zero focus steal" and DO trigger Ink's submit (because injection happens at the PTY master, like real input). This is exactly what Anthropic's Agent Teams uses under the hood. Best practices: literal mode `-l`, ~0.3–0.5s delay before Enter, strip ANSI from `capture-pane`, and use a `.ready` handshake to only send when the worker is idle.
- On **Windows 11**: there is **no native tmux**. The closest equivalent is **`psmux`** (github.com/psmux/psmux), a native multiplexer of **25,600 lines of Rust implementing 76 tmux commands via Windows ConPTY directly (no WSL/Cygwin)**, installable via winget/scoop/choco/cargo — so `tmux send-keys` through psmux injects at PTY level the same way real tmux does. Notably, **psmux now handles Claude Code integration natively**: per its `docs/claude-code.md`, it offers a config option (`set -g claude-code-fix-tty on`), always exports `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS=1`, injects `--teammate-mode tmux` into every `claude` invocation, and requires PowerShell 7+ (this supersedes the earlier need for a manual version-string shim like the `gonnector/claude-psmux-team` helper repo). Remaining gotchas: sandbox/path bugs when launching from the home directory (#43840, #42848). Anthropic declined to officially support this route (#34150 closed as not planned). `wmux` (github.com/openwong2kim/wmux) is a Windows multiplexer too but does **not** expose tmux send-keys — it only does MCP-based agent messaging.
- **AutoHotkey `Send`/`SendText`** and **PowerShell `System.Windows.Forms.SendKeys`** can deliver "text + Enter" into a Claude Code CLI window and, because they generate physical-style input, can trigger submit — **but only into the focused/foreground window** (they steal focus), and AHK's background `ControlSend` is unreliable against ConPTY terminals. Special characters (`^ + ! # { }`) must be escaped.
- **VSCode integrated terminal** (running `claude` as a CLI in the terminal, not the panel) is injectable via `workbench.action.terminal.sendSequence` — but plain `\r`/`\n` still won't submit; you need the bracketed-paste trick. The **VSCode extension chat panel (sparkle icon)** is a webview contenteditable that ignores keystroke-injection APIs entirely; the feature request for `claude-vscode.sendMessage` (**#27873**) was closed unimplemented.

**(C) Headless invocation (spawn a fresh turn).** `claude -p "<msg>" --resume <session_id>` (or `--continue`) reliably delivers a message and even resumes prior context — but it runs as a separate non-interactive invocation that **exits after the response**; it does not inject into the live interactive session. Streaming JSON input (`--input-format stream-json` + `--output-format stream-json`) allows multi-turn input to a *headless* process you control, which is the basis for "wrapper daemon" designs. On Windows, writing to a node-pty/pywinpty-wrapped `claude.exe`'s stdin echoes text but is **not consumed** by Claude's prompt/permission handlers (#38299, #36156), so a persistent interactive wrapper is unreliable on Windows; headless `claude -p` is the dependable programmatic path there.

### 3. Anthropic's official Agent Teams partly solves this — with platform limits
Anthropic shipped **Agent Teams** ("swarm mode") as a **research preview on February 5, 2026 in Claude Code v2.1.32, alongside the Opus 4.6 model release**. Per Anthropic's "Introducing Claude Opus 4.6": *"We've introduced agent teams in Claude Code as a research preview. You can now spin up multiple agents that work in parallel as a team and coordinate autonomously on shared codebases."* It is enabled via `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS` in settings.json and provides exactly the primitives you want: a shared task list (`~/.claude/tasks/{team-name}/`), inter-agent "inbox" messaging, **automatic message delivery** (*"when teammates send messages, they're delivered automatically to recipients. The lead doesn't need to poll for updates"* — code.claude.com/docs/en/agent-teams), idle notifications, and `TeammateIdle`/`TaskCompleted` hooks. Critical limitations for your setup:
- **Split-pane mode requires tmux or iTerm2 and is explicitly NOT supported in VS Code's integrated terminal, Windows Terminal, or Ghostty.** The default "in-process" mode works everywhere but runs teammates inside one Node process/terminal (less visibility) and **does not survive `/resume`**.
- **On Windows, split-pane teammates fall back to in-process** because, per issue #34150, *"process.stdout.isTTY is undefined under Bun SFE on Windows, which forces in-process mode regardless of teammateMode: 'tmux' settings"* (tracked in #26244); Anthropic closed #34150 (psmux support) as not planned.
- It is one-team-per-lead, experimental, with known session-resumption and task-status-lag issues. The official docs give **no fixed token multiplier**, only: *"Agent teams use significantly more tokens than a single session. Each teammate has its own context window, and token usage scales with the number of active teammates."* (Community reports put a 3-teammate/30-minute team at roughly 3–4× a single session; treat that as anecdotal, not official.)
- A subscription-policy change — **announced Friday April 3, 2026 by Boris Cherny (Head of Claude Code) and enforced Saturday April 4, 2026 at 12:00 PM PT, beginning with OpenClaw (247K+ GitHub stars)** — blocked Claude **Pro/Max subscribers** from using subscriptions with most third-party orchestration frameworks. Cherny: *"We've been working hard to meet the increase in demand for Claude, and our subscriptions weren't built for the usage patterns of these third-party tools."* Anthropic offered a one-time credit equal to one month's subscription (redeemable by April 17, 2026). **Claude Code on Pro/Max and API-key/Bedrock/Vertex users are unaffected**, and official Agent Teams remains allowed — making it the safest supported orchestration path if you're on Pro/Max.

### 4. Community projects that implement inter-agent messaging for Claude Code
- **Walkie-Talkie** (suruseas, GitHub + DEV post): "Slack for Claude Code instances." Architecture: `Claude Code A ──stdio──> MCP Server ──HTTP──> Hub ──HTTP──> MCP Server ──stdio──> Claude Code B`. Uses **HTTP long-polling** (MCP server holds connection up to ~1 hour) for a real-time feel over MCP stdio transport. The Hub is "just an HTTP server," so any cron/script/`curl` can push a message to a connected agent. Security warning from the author: the SKILL.md makes agents *execute* operator messages with full tool access — **never expose the Hub to the internet**.
- **agent-room / Stop-hook pattern** (DEV): the 15-line blocked-Stop-hook + Redis long-poll design described above — "no polling, no sidecar processes, no changing Claude Code itself."
- **claude-squad** (smtg-ai): zero-setup tmux-based multi-agent terminal UI (`cs`); manages multiple agents in separate workspaces/git worktrees.
- **AgentSpeak / claude-teams-language-protocol** (yuvalsuede): a token-efficient inter-agent message format that integrates with native Agent Teams, cutting messaging tokens 60–70%.
- **agent-chat-gateway** (HammerMei), **claude-agent-chat** (snedea), **AgentsRoom**: chat-room/remote-control layers, mostly orchestrator-driven or WebSocket-relay based.
- **EPAM's reference team**: 6 Claude Code roles on a single Max subscription coordinating through a **SQLite message queue ("taskbox")** + GitHub Issues, with a supervisor that polls in the background and uses tmux `send-keys` to dispatch — a close architectural analog to what you're building.

### 4a. "A2A"-named GitHub projects for Claude — two distinct categories
Searching specifically for **A2A (Agent2Agent)** + Claude surfaces a set of repos, but they split into two groups that mean very different things. Critically, the projects that carry the *formal* A2A-protocol branding are mostly the wrong tool for a single-machine local team, while the one that is genuinely on-point only *borrows* the "agent-to-agent" name and is **not** the formal protocol.

**Category 1 — Formal A2A-protocol wrappers (interoperability-oriented; likely overkill here).** These implement the open **Agent2Agent protocol** (github.com/a2aproject/A2A, now under the Linux Foundation): JSON-RPC, published **AgentCards**, `message/stream` envelopes. They expose Claude Code as a standards-compliant A2A *server endpoint* for cross-vendor / cross-network interop, treating Claude as headless rather than as your live interactive sessions:
- **`ericabouaf/claude-a2a`** — A2A server wrapper around the Claude Code SDK; runs on `localhost:3008`, serves an AgentCard; explicitly flagged "not production ready."
- **`dwmkerr/claude-code-agent`** — runs Claude Code as an *isolated, containerized* A2A agent you address by `curl`-ing JSON-RPC `message/stream` envelopes at `localhost:2222`; Docker-oriented, heaviest option, strong operation/message visibility.
- **`hybroai/a2a-adapter`** (PyPI) — SDK that converts Claude Code (plus LangGraph, CrewAI, LangChain, Codex, Ollama) into A2A-compatible servers with auto-generated AgentCards and streaming.
- **`caomyer/claude-code-a2a-multiagent`** — uses the formal A2A protocol *plus* tmux to drive visible Claude Code CLI panes (frontend / backend / PM / UX roles + host orchestrator). Promising design but early scaffolding (~Phase 1 of 7 complete) — a reference, not something to run yet.

For a single dev with 4–5 local agents, the formal A2A spec is designed for cross-vendor, cross-network interop and adds JSON-RPC/AgentCard ceremony you don't need on one box.

**Category 2 — `dopatraman/a2a` (on-point; *not* the formal protocol).** This repo reuses the "agent-to-agent" name but implements exactly the **MCP-bridge + hub + push-injection** pattern this report recommends. Architecture:

```
Claude A --stdio--> MCP Client A --WebSocket--> Hub (:7800) <--WebSocket-- MCP Client B --stdio--> Claude B
```

A per-agent MCP client bridges stdio↔WebSocket; **when the hub pushes an event, the MCP client injects it into Claude's session as a `<channel>` notification** — the same delivery mechanism as Walkie-Talkie, but with a cleaner **watch/emit** model. Agents call `connect(name)`, `watch(agent_id)`, and `emit(content)`; the hub routes each event only to that emitter's watchers. Tools provided automatically: `connect`, `disconnect`, `watch`, `unwatch`, `emit`, `list_agents`. The canonical use case is one agent running code while another *watches* and debugs live.

Caveats:
1. Requires launching with `claude --dangerously-load-development-channels`, an experimental gated flag (real adoption risk).
2. Very early — 6 stars, 8 commits, 39 tests, no releases; **direct/targeted messaging, message queues, auth, and backpressure are all still on the POST_MVP roadmap** (today it is pub/sub-by-watching only, with no addressed DMs).
3. It rides the MCP long-poll path, so it does **not** defeat the idle-agent problem — injection still depends on the session consuming the `<channel>` notification.

**Why it matters for you:** `dopatraman/a2a` is the cleanest readable reference implementation of the recommended pattern, and its `watch`/`emit` envelope is a nicer primitive than raw broadcast for a role-based team. Its one structural gap is **targeted, to-/from-addressed delivery** — which is precisely what your existing file-header queue already does well. So the natural fit is: lift its MCP-bridge + hub design for the fast path, and keep your file queue as the addressed-delivery + offline-fallback layer (rather than waiting on its roadmap for DMs).

### 5. Lightweight local broker comparison (single machine, single dev)
| Option | Footprint / setup | Push model | Persistence | Windows | Best fit here |
|---|---|---|---|---|---|
| **Redis pub/sub + lists** | Single server; ubiquitous clients; on Windows via Memurai/WSL/Docker | Pub/sub (fire-and-forget) + blocking list pops (`BRPOPLPUSH`) for long-poll | Optional (RDB/AOF); pub/sub itself ephemeral | Native build deprecated; use Memurai/Docker | **Top pick** — the blocking-list long-poll maps perfectly onto the Stop-hook wake loop |
| **NATS** | Single ~20MB Go binary, trivial to run; one binary also speaks MQTT | Subjects (pub/sub), request/reply, "no responders" instant feedback; JetStream adds durable streams + delayed delivery | JetStream (durable) | Single binary runs natively | **Strong pick** — if you want durable streams + presence-style "no responders" detection in one binary |
| **MQTT (Mosquitto)** | Small broker; retained messages + LWT (last-will) ideal for presence | Topic pub/sub; QoS levels; retained msgs | Limited (retained/persistent session) | Native | Good if you like retained-message presence + last-will offline detection |
| **ZeroMQ / nanomsg** | No broker (brokerless library) | Brokerless push/pull, pub/sub | None | Cross-platform lib | Lower-level; more code, no central state |
| **Named pipes / Unix sockets / FIFOs** | OS-native, zero install | Point-to-point | None | Windows named pipes ≠ Unix sockets (need adapter) | Simplest for 2 processes; awkward for dynamic N-agent membership |

For your case (4–5 dynamic agents, file-queue fallback already built), **Redis or NATS** hit the sweet spot: one process to run, trivial clients in any hook language, and a blocking-read primitive that the Stop hook can long-poll. MQTT is a fine alternative if you specifically want **retained messages + last-will-and-testament** for presence/offline detection out of the box.

### 6. Presence / availability detection and the file-queue fallback
- **Heartbeat + lock/PID files**: each agent's sidecar writes a heartbeat (timestamp) to a known path or broker key every N seconds; "available" = fresh heartbeat, "offline" = stale/missing. Stale PID-file detection (process named by PID no longer exists ⇒ lock invalid) is a well-established pattern; libraries like Python `pid` handle stale detection and locking.
- **Connection-state presence**: with a broker, presence falls out naturally — NATS reports "no responders" instantly when nobody is subscribed to a subject; MQTT's **last-will-and-testament** publishes an "offline" message automatically on disconnect; a WebSocket/long-poll connection's open/close state is itself the presence signal.
- **Clean fallback logic**: the *sender's* sidecar (not the agent) decides: try to deliver to the target's live channel/heartbeat; if the target is offline or delivery isn't acknowledged within a timeout, **write the message into your existing file queue** (with the `to/from/subject/topic` headers you already use). When the target next starts or submits a prompt, its `SessionStart`/`UserPromptSubmit` hook drains the file queue — your current mechanism — so nothing is lost. This makes the file queue the durable backstop and the broker the fast path.

## Details — Recommended architectures, simplest → most robust

**Option 0 (baseline you already have):** File queue drained by a hook on human interaction. Reliable, zero push. Keep it as the fallback layer in every option below.

**Option 1 — Stop-hook self-wake + local broker (recommended starting point; fully cross-platform, no keystroke injection).**
- Run one **Redis** (or NATS) locally. Each agent gets a per-agent inbox key/subject.
- Each agent's `settings.json` registers: a `SessionStart`/`UserPromptSubmit` hook that drains both the broker inbox and the file queue and injects via stdout/`additionalContext`; and a **`Stop` hook that blocking-pops the inbox for ~25–30s**, returning `decision: block` + the message if one arrives, else exiting 0 (honoring `stop_hook_active`, capping iterations).
- A **sender-side sidecar** publishes to the target's inbox; if the target's heartbeat is stale, it writes to the file queue instead.
- **Pros:** works identically on Windows/Mac/Linux and in both CLI and VSCode (the panel runs hooks too); no fragile keystroke injection; integrates your file queue as the natural fallback. **Cons:** "push" is bounded by the Stop-hook poll window (near-real-time only while the agent is between turns or kept alive); idle-too-long agents still rely on the file-queue drain on next activity; watch token/turn budget.

**Option 2 — Add true keystroke push for "asleep" agents (Mac/Linux, or Windows+psmux).**
- Layer a **per-agent or central supervisor daemon** that subscribes to the bus. When a message targets an agent that is fully idle (Stop already returned), the supervisor delivers it by **`tmux send-keys`** (Mac/Linux/iTerm2) or **psmux send-keys** (Windows) into that agent's pane, using literal mode + delayed Enter + a readiness handshake. If the pane isn't found/healthy, fall back to the file queue.
- **Pros:** genuine wake-from-idle without burning turns in a poll loop. **Cons:** OS-specific; Windows requires psmux (PowerShell 7+); brittle vs. VSCode panel (use the integrated terminal with bracketed-paste, or keep VSCode agents on Option 1 only).

**Option 3 — Adopt Anthropic Agent Teams (most "native," best if you can standardize the environment).**
- Enable `CLAUDE_CODE_EXPERIMENTAL_AGENT_TEAMS`; let a lead spawn teammates. Automatic message delivery + shared task list gives you push for free; use `TeammateIdle` to auto-assign queued work and `TaskCompleted` as quality gates. Keep the file queue as the cross-team / cross-restart fallback (teams don't survive `/resume`).
- **Pros:** officially supported, no custom broker, peer messaging built in. **Cons:** VSCode panel and Windows Terminal can't do split panes (in-process only on Windows); experimental; significantly higher token use; one team per lead.

**Option 4 — Headless wrapper / supervisor (most control, most code).**
- A central orchestrator keeps session IDs and uses `claude -p --resume <id>` (or streaming-JSON input) to push tasks; presence via heartbeats; file queue for offline. Best on Mac/Linux; on Windows use headless `claude -p` (interactive PTY input injection is unreliable per #38299).

## Recommendations (staged, with thresholds)
1. **Now (1–2 hrs):** Stand up **Redis** (Memurai or Docker on Windows) and implement **Option 1** for all agents. This gives near-real-time delivery for active agents and keeps your file queue as the guaranteed fallback. Benchmark: messages delivered to an active agent within one Stop-poll window (~30s), zero lost messages to offline agents.
2. **Read `dopatraman/a2a` before you build (½ hr):** It is the cleanest reference for the MCP-bridge + hub + `<channel>`-injection pattern. Lift its hub routing and MCP-client bridge design; keep your file-header queue as the addressed-delivery layer it lacks. (Don't depend on its `--dangerously-load-development-channels` requirement in production.)
3. **Add presence (½ day):** Have each agent's sidecar heartbeat to a Redis key every 10–15s; sender checks freshness and routes to file queue if stale. Threshold to proceed: you can reliably distinguish online/busy/offline.
4. **If you need wake-from-idle (not just between-turns):** On Mac/Linux, add **Option 2** with tmux send-keys. On Windows, evaluate **psmux** (which now self-configures Claude Code integration); if psmux proves brittle in your sandbox/path config, stay on Option 1 + accept file-queue-on-next-activity for fully idle agents. Threshold: only invest here if agents routinely sit idle long enough that the file-queue drain delay actually hurts.
5. **If you can standardize agents onto Mac/Linux (or psmux) and are on Pro/Max:** pilot **Agent Teams** for a single feature; if its automatic delivery + task list meet your needs, it can replace much of your custom broker plumbing. Switch back if the experimental limitations (no `/resume`, token cost, one-team-per-lead) bite.
6. **Keep the VSCode panel agents on hook-based delivery only** — do not attempt panel injection; if a VSCode agent needs true push, run `claude` in its **integrated terminal** instead of the panel.

## Caveats
- **Everything here targets a fast-moving, partly-experimental surface.** Hook event names/behaviors (TeammateIdle, FileChanged, async hooks, bracketed-paste behavior) and Agent Teams are evolving across Claude Code v2.1.x releases; verify against the current `code.claude.com/docs/en/hooks` and `/agent-teams` before building.
- **Stop-hook loops are genuinely dangerous to your token/session budget** (#55754). Always gate on `stop_hook_active`, cap iterations, and prefer short poll windows.
- **The April 4, 2026 subscription policy** restricts Pro/Max use with third-party orchestrators; a *self-built* broker + hooks for your own agents is different from a packaged third-party framework, but if you're on Pro/Max, prefer official Agent Teams or API billing for anything resembling a framework, and monitor Anthropic's terms.
- **Windows is a second-class citizen** for multi-agent Claude Code: no native tmux, VSCode panel injection impossible, interactive PTY stdin injection unreliable. Plan around the hook-based path as the durable core and treat keystroke push as a Mac/Linux/psmux enhancement.
- **Security:** any "execute inbound messages as tasks" design (Walkie-Talkie- or `dopatraman/a2a`-style) is a remote-code-execution surface. Keep brokers/hubs bound to localhost, never expose them to the network, and consider authenticating messages between your own agents.
- The strongest community consensus is the Stop-hook + bus pattern plus tmux/psmux send-keys for wake-from-idle; the "A2A"-named repos are useful reference implementations of the MCP-bridge half of that, not a turnkey addressed-messaging solution for a local 4–5-agent team.
