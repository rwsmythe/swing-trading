## 2. The defect Reviewer B found — VERIFIED real
The fix releases the client by clearing `app.state.schwab_client` + `del` local + `gc.collect()`. In production that is NOT the only strong reference. `swing/web/app.py:302-404` `_install_web_marketdata_caches` builds the client (`:314`) and installs two ladder-fetcher **closures that close over the same `client`**:
- `_quote_hook` → `schwab_client=client` (`app.py:351`), stored via `price_cache.set_ladder_fetcher` (`:402`)
- `_bars_hook` → `schwab_client=client` (`app.py:379`), stored via `ohlcv_cache.set_ladder_bars_fetcher` (`:403`)

Both caches live on `app.state` for the app's lifetime → the client has **three** strong refs when the ladder is active. The release clears one → `gc.collect()` cannot finalize the client → the `BEGIN EXCLUSIVE` tokens-DB handle stays open → `os.replace` still raises `[WinError 32]`. The bug is unfixed in the exact (ladder-active) scenario it occurs in. (Secondary latent issue: even if rename somehow succeeded, the two closures keep referencing the OLD, rotated-token client after reconstruct.)

## 3. Why A and the regression test BOTH missed it (the important part)
**This is a METHODOLOGY result, not just "B's model is better."** The closure-installing code (`_install_web_marketdata_caches`) is NOT in the diff. Reviewer A was prompted to review the inline diff ONLY ("do NOT run git, do NOT read other files — everything you need is inline") — so A was **structurally blinded** to a defect that depends on un-changed surrounding code (the reference graph / closures). Reviewer B (codex-auto-review) ran with repo access — it grepped `schwab_client` across `swing/web/app.py` + the wider tree and read the factory — which is how it found the closure refs. Both the **methodology** (diff-only vs repo-access) and the **model/mode** differ; the methodology is the load-bearing difference here.

The regression test fell into a variant of the synthetic-vs-production gotcha the brief §4 explicitly warned about: it exercises the POST path but with a bare `_StubLockingClient` injected straight into `app.state` (one reference, no cache-hook closures, because the test app has no creds → ladder inactive → factory returns None). So drop+gc finalizes the stub and the test goes green — it does not replicate the production reference graph. **Banked lesson candidate:** a release/finalization test MUST replicate the FULL production reference graph (every closure/cache/caller that holds the resource), not just the one reference the brief named.
