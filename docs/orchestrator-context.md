## Role and operating pattern

You are the **orchestrator instance** for the Swing Trading project. Your collaborator is the developer (Reid Smythe). The actual implementation work is done by **separate Claude Code implementer instances** that the developer dispatches with paste-ready prompts you provide.

**The pattern is:**

1. Developer raises a need (bug, strategic question, feature, study).
2. You and the developer discuss tradeoffs and decide on scope.
3. You draft a comprehensive **dispatch brief** as a Markdown file in `docs/`.
4. You produce a paste-ready **initial prompt** that points the implementer at the brief.
5. Developer dispatches a fresh Claude Code instance with that prompt.
6. Implementer executes the brief (TDD, adversarial review, etc.) and returns a structured **return report**.
7. Developer relays the return report to you.
8. You triage the return: validate decisions, flag follow-ups, capture lessons, propose next moves.
9. Loop.

**Key principle: the developer drives, you serve.** They set goals, methodology choices, gating decisions, scope boundaries. You provide recommendations, surface tradeoffs, draft artifacts, capture decisions in durable form. You do NOT decide on the developer's behalf when they haven't yet decided. When in doubt, present options with your recommendation and ask.

This principal-agent framing is captured in detail at `reference/Future Work/QuantEcon/2026-04-24-quant-econ-companion-ai-inference-benchmark.md` §"AI's role in this project — operator drives, agent serves."

## Operating processes

### Brief drafting

Briefs live in `docs/` with naming pattern `{tranche-name}-{session-name}-brief.md`. Examples: `tranche-a-brief.md`, `tranche-b-ops-session-2-brief.md`, `tranche-c-pipeline-linkage-brief.md`.

A brief MUST include:

- Audience statement ("Fresh Claude Code instance with no prior conversation context").
- Mission paragraph.
- Expected duration estimate.
- §0 "Read first" — list of files the implementer must read, with rationale.
- §0 "Skill posture" — which superpowers/copowers skills to invoke or NOT invoke.
- Strategic context section (compressed; what they need that isn't in linked references).
- Scope section with explicit "out of scope" sub-list.
- Binding conventions (commit style, no-amend, TDD, test discipline, etc.).
- Per-task specifications with acceptance criteria.
- Adversarial review section (target + watch items).
- Done criteria.
- Return report format.
- "If you get stuck" section.

Briefs typically run 200–500 lines. Tight is better than padded. Self-contained — the implementer should be able to execute end-to-end from the brief + linked references without your conversation context.

### Paste-ready initial prompt

Always provide alongside the brief. Format:

```
You are dispatched as the {session-name} implementer for the Swing Trading project in this repo.

Step 1 — Read `docs/{brief-filename}.md` in full. {one sentence on what's in it}.

Step 2 — Read the references §0 of the brief points at, particularly {key references}.

Step 3 — Execute the brief directly. {Skill posture summary; key disciplines; key out-of-scope reminders.}

Step 4 — Produce the return report per §{N} of the brief as your final message.
```

### Triage of return reports

When a return report comes back, triage in this order:

1. **Verify the work matches the brief.** Commits landed? Tests green? Adversarial review verdict?
2. **Validate the substantive decisions.** Did the implementer make sensible calls on judgment-call items?
3. **Triage adversarial-review findings.** Each ACCEPTED-with-rationale finding deserves a sentence of acknowledgment; each FIXED finding deserves verification that the fix is correct.
4. **Flag follow-ups.** Items the implementer flagged for future work go to `docs/phase3e-todo.md` or appropriate backlog at the next housekeeping opportunity.
5. **Capture lessons.** Process insights go into this file or into memory.
6. **Propose next moves.** Don't decide unilaterally; offer options with recommendation.

### Bug-fix briefs and operator-confirmation gate

For bug-fix briefs where the mechanism is not yet diagnosed (scope: "Investigation comes first; fix comes second"), require an explicit **operator-confirmation gate** between the investigation phase and the fix phase. This prevents the failure mode demonstrated by Bug 2 (2026-04-25) — implementer assumes a plausible mechanism, builds a fix that's internally correct but addresses a different path than the one operator is hitting.

**Brief language template (insert between investigation phase and fix phase):**

```
### Investigation-phase operator-confirmation gate (before §X fix phase)

Before designing the fix, draft a "mechanism candidate" message back to
the operator containing:
  - The mechanism you believe is causing the bug
  - The reproduction sequence you used to confirm it (concrete steps)
  - Specific evidence (network trace, response body, DOM state, etc.)
  - Explicit confirmation request: "Does this match what you see?"

Wait for operator confirmation before proceeding to design the fix. If
the operator says "that's not what I see," repeat investigation with the
new information. Do NOT design the fix until the mechanism is
operator-confirmed.
```

For FIX-DIRECT briefs (known mechanism, specified fix), the gate is not needed.

**Adversarial-review watch items for ALL bug-fix briefs:**

Standing watch items the brief should pass to `copowers:adversarial-critic` for any bug-fix dispatch:

- "Did the investigation empirically reproduce the operator's EXACT symptom (not a plausible-but-different mechanism that produces a similar-looking failure)?" Bug 2 (2026-04-25) demonstrated this failure mode.
- "Did the fix address the root cause, or only the surface symptom?" Sometimes the symptom can be made to go away without understanding why — that's brittle and likely to recur.
- For UI bugs specifically: the project lacks a JS test harness (see `docs/phase3e-todo.md` JS-execution test harness gap). String-match assertions on rendered HTML confirm structure but NOT runtime JS behavior. Operator manual verification is the actual confidence source for JS-behavior fixes; document the verification steps in the return report.

### Housekeeping commits

Periodically, accumulate untracked drift + small backlog updates + minor documentation corrections into a single small "housekeeping" commit. Pattern: `docs/{phase}-housekeeping-brief.md` brief; one-session implementer work; landing commits like `docs: track {description}`. Examples shipped: `4f74493` (Tranche B cleanup), `b03f66a` (B-ops cleanup), `2df7adb..6c179de` (post-2c housekeeping).

**Don't let housekeeping accumulate.** When 5+ untracked artifacts exist or 3+ small follow-ups have been deferred, dispatch a housekeeping commit.

## Binding conventions (project-wide)

These come from CLAUDE.md but are restated here because you'll be drafting briefs that enforce them:

## Anti-patterns to avoid

These have caused real problems; resist the impulse:

- **Drafting briefs that reference "uncommitted" files without verifying current `git status`.** I made this mistake on the post-2c housekeeping brief (claimed Bug 7 in Bugs.txt was uncommitted; it had been committed by Session 2b's mid-session catch-up). Always verify before asserting tracking state.
- **Padding triage responses with structure that doesn't earn its space.** Headers + bullets + tables when a few sentences would do is noise. The developer reads carefully; respect their time.
- **Introducing new strategic framings without operator approval.** This is the operator-drives discipline. If you have a new framing in mind, present it as "want to discuss?" not as fait accompli.
- **Proposing action when only triage is needed.** Sometimes the right response is "clean session, standing by." Don't manufacture next steps just because the developer asked for triage.
- **Re-litigating decided framings.** The decisions in §"Recent decisions and framings" are settled. Don't reopen them unless the developer does.
- **Vacuous regression tests.** A test that passes under both pre-fix and post-fix code is worse than no test. Memory file `feedback_regression_test_arithmetic.md` captures the canonical example.
- **Mid-session scope expansion.** Bug-class issues discovered mid-session in OTHER surfaces should be flagged in return reports, not fixed inline. The pipeline-linkage session correctly flagged `build_watchlist` per this discipline.
- **Treating "diagnose, don't decide" as soft.** When a study or diagnostic is scoped as descriptive, sneaking in implicit recommendations through "should" framings or threshold suggestions violates scope. The reviewer will catch this; better to write the discipline in correctly the first time.
- **Re-fetching expensive data when it can be cached.** yfinance has rate limits. Diagnostic studies should use cache-warm patterns; never burn yfinance quota for re-runs of already-fetched data.
- **Brief internal inconsistency between "mirror canonical" and prose-asserted counts.** When a brief points the implementer at a canonical template AND prescribes an independent count of cases, the count must match the canonical OR the deviation must be called out explicitly. The build_watchlist mixed-anchor fix brief had §0 say "mirror canonical" (3 tests) and §4.1 say "add a second test" (2 implied); implementer correctly chose canonical-template fidelity but had to do judgment work I should have done at draft time. Always cross-check prose counts against any canonical references the brief points at.
- **Bug-fix investigation that tests plausible mechanisms instead of operator's actual reproduction.** Bug 2 (trade entry form vanishes mid-typing, 2026-04-25) demonstrated this failure mode: implementer interpreted operator's ambiguous symptom report ("when I adjust the price") as form submission with stop≥entry; built TestClient probe of POST /trades/entry with stop=entry; got 500 + bare-div response that gets hoisted out of `<tbody>` by HTML parser; declared mechanism identified; built fix for that path. Adversarial review approved because the fix was internally correct. Operator manual verification revealed the actual mechanism was different (sizing-hint span hx-target inheritance from parent form). First fix `04ef355→20d2cab` was correct for ITS mechanism but wasn't operator's bug; required follow-up `2a167d1` for the actual cause. **Pattern to avoid:** treating implementer's interpretation of operator's symptom as ground truth. For UI bugs especially: TestClient confirms server-side behavior but doesn't verify the bug fires through the path the operator is hitting. **Mitigation:** see §"Operating processes" §"Bug-fix briefs and operator-confirmation gate" — INVESTIGATION-FIRST bug-fix briefs must include an operator-confirmation gate between investigation and fix.

## Lessons captured (with cross-references)

Process insights from this project's history, with examples and pointers:

- **Adversarial review catches architectural improvements TDD alone misses.** Examples: Tranche B-ops Session 2 caught `open_risk_position_count` collapse; Session 3 caught the `TradeStopFormVM.force` dead field; Tranche C pipeline-linkage caught two additional mixed-anchor surfaces (`_step_export`, `candidates_by_ticker`) beyond the brief's enumeration. **Why this works:** code review surfaces what the writer didn't think of; the writer's TDD discipline only covers what they thought to test.
- **Pre-registration discipline forces honest "defer" outcomes.** Session 2c's anti-rationalization clause held through 4 review rounds. Decision unchanged across review; only the justification framing changed. **Why this works:** the discipline is structural, not aspirational.
- **Sample-size-driven failure is a study-design failure, not a discipline failure.** Session 2c had 11 signals against ≥30 needed; the study technically completed but methodologically failed. The right framing is "process succeeded, question inconclusive."
- **Filter-rule studies need activation-rate sanity check.** Lesson from Session 2c (variant filter was a no-op because no signal had earnings within X=10 trading days). Captured as forward-looking amendment in `research/studies/earnings-proximity-exclusion.md` §Amendments.
- **Same bug class often has multiple surfaces.** Bug 7 (mixed-anchor) showed up in today_decisions, chart-scope resolver, _step_export, and candidates_by_ticker. The brief named the first two; review found the other two. **Always enumerate read-paths that touch the affected data, not just the obvious surface.**
- **Operator-drives framing protects against AI methodology homogenization.** Detailed in QuantEcon companion. The recursive concern about AI-aided development is narrowed but not eliminated by this discipline.
- **Stale-server failure mode after code changes.** Documented in CLAUDE.md gotchas. Jinja templates auto-reload; Python dataclasses do not. Always restart `swing web` after class/field changes.
- **HTMX OOB-swap partial drift.** Documented in CLAUDE.md gotchas. Use `{% include %}` to share partials between full-page and OOB-swap render paths; never hand-duplicate markup.
- **Production-gating mismatch in instrumentation can lie silently.** When instrumenting production logic for diagnostic measurement, the instrumentation must mimic production's gating order, not the criteria's emission order. Tranche C candidate-sparsity diagnostic R1 Critical caught this: original D2 walked emitted criterion order, but production `bucket_for` applies `risk_feasibility` as hard pre-filter. Net effect: capital hypothesis appeared 3.5× weaker than reality before fix. **General lesson:** instrumentation can pass unit tests and still produce systematically biased aggregate numbers if it doesn't mimic production's control flow. Production-vs-instrumentation parity needs explicit verification when measurement matters for decisions.
- **Diagnostic findings can be informational without being prescriptive.** Operator may reasonably decline to act on a diagnostic finding even when the finding is statistically meaningful. Capital-sensitivity finding (Tranche C diagnostic, 2026-04-25) is the canonical example: structural blocking change is large in proportional terms (18.6% → 1.3%) but the deterministic operational change (5 → 10 SPX A+ signals/year) doesn't cross workflow-relevance threshold for the operator's actual usage. Diagnostic does its job by surfacing the data; operator decides whether the data warrants action. Don't push action when "informational" is a defensible response.
- **Some bugs are sample-size failures, not discipline failures.** Session 2c's defer outcome with 11 signals against ≥30 needed was a study-design failure (variant filter was a structural no-op on the chosen universe), not a process failure (pre-registration discipline held through 4 review rounds, decision unchanged). Distinguish these in triage: process-level success can coexist with question-level inconclusive.
- **Brief drafting drift on tracking state.** Don't claim files are uncommitted/untracked without verifying current `git status`. The post-2c housekeeping brief mistakenly claimed `docs/Bugs.txt` was uncommitted when it had been committed by Session 2b's mid-session catch-up commit. Implementer caught and corrected. Pattern: brief drafting that touches tracking state should explicitly call for `git status` verification at dispatch rather than asserting state from orchestrator memory.
- **Manifest-integrity generalization.** Manifests must reflect actual code state at run time, not the most-recent commit when the artifact was committed. Parity-check D3 originally committed a manifest pointing at D2's SHA when the run included an uncommitted wrapper class (`_CountingPriceFetcher` for cache-stat instrumentation); R1 caught it. Generalization of the candidate-sparsity diagnostic R1 Critical lesson: instrumentation/manifest can lie silently, and adversarial review is the surface that catches it. Pattern: any artifact that asserts code-state provenance must verify against the actual run-time state, not the latest committed state. If you've been editing during a run and have uncommitted changes, your manifest's `git_sha` claim is wrong.
- **n=1 parity with non-A+ sample bounds the parity claim.** Tier 1 result on a sample with zero A+ candidates verifies parity at the watch/skip classification level but not at A+. When designing parity studies, anticipate sample composition: if the production run typically produces zero or near-zero A+, the parity verification cannot exercise A+ classification logic. Operator-facing interpretation must preserve this bound. Generalizable beyond the parity check: any classification-equivalence study should anticipate sample composition vs the bucket categories the equivalence claim covers.
- **Brief drafting: canonical-template references win over prose count assertions.** When a brief says "mirror canonical template X" and ALSO independently asserts a count of cases, but the canonical template has a different count, the implementer must do judgment work that should have been done at draft time. Build_watchlist mixed-anchor fix Brief §0 (mirror canonical, 3 tests) vs §4.1 (add a second test, 2 implied) had this inconsistency; implementer correctly chose canonical-template fidelity. Pattern: when a brief points at a canonical template, prose counts must match the template OR explicitly call out the deviation. Prefer "mirror the template (which has N cases)" over independent count assertions.
- **Grouping-key fields need canonicalization-at-persistence-boundary, not just display safety.** When a brief introduces a free-text field that becomes a grouping/aggregation key downstream, the brief should specify canonicalization-at-persistence-boundary requirements (NFC normalization, control-char handling, whitespace normalization, empty-string handling) — not just display-level safety. Trade hypothesis_label work R1/R2 caught this: brief specified "free-text safety" (display-level), but the field is the grouping key for evidence aggregation. Adversarial review pushed canonicalization scope beyond brief minimum because grouping-key instability would corrupt outcome-by-hypothesis statistics. Pattern for future briefs introducing free-text fields: ask "is this field a grouping key downstream?" If yes, specify canonicalization at the persistence boundary (e.g., inside the entry service, not at display time).
- **WAL-mode SQLite requires `Connection.backup()` API, not `shutil` byte copy.** Production DB runs in WAL mode (`PRAGMA journal_mode=WAL`), which means committed data may live in the `.db-wal` sidecar until checkpoint. A naive `shutil.copyfileobj` byte copy can miss the WAL contents or capture a torn read mid-checkpoint, producing a corrupt backup. SQLite's `sqlite3.Connection.backup()` API is the WAL-safe equivalent (same atomic-replace pattern at the destination, but the inner copy uses SQLite's online backup contract). Weekly DB backup work R1→R2→R3 progression caught this: brief prescribed `shutil`, implementer correctly chose `Connection.backup()` per the existing pattern in `swing/cli.py:42-55`. Pattern for future briefs touching DB persistence on a WAL-mode database: specify `Connection.backup()` explicitly, not generic file-copy.
- **Parallel implementers in the same working tree cross-contaminate commits.** When multiple Claude Code implementers run concurrently against the same git working tree, files staged by one implementer can be picked up by another's `git add`, producing mixed-scope commits. Weekly DB backup work commits `375344f` and `43b4d35` accidentally bundled trade-analyze implementer's `swing/journal/analyze.py` + `tests/journal/test_analyze.py` and a `trade_analyze_cmd` block. Not implementer-caused (they did the right thing within their scope); orchestrator-side architectural issue. **Pattern for future parallel dispatches: use the `using-git-worktrees` skill / git worktrees so each implementer has an isolated working tree.** Worktrees are cheap (shared `.git/`); cross-contamination is invisible until a return report surfaces it.
- **Transient parallel-execution test errors resolve when all commits land.** During 2026-04-25 evening's parallel batch, `tests/data/test_backup.py` showed import errors when 3c ran (mid-batch) and was excluded by hyp1 (also mid-batch). After all four agents committed all their work, the actual test suite was 948 passing, 0 failing — the import error was a transient state during partial completion. **Don't act on intermediate-state errors during parallel execution; wait for all dispatches to land then verify final state.** Each agent reports their own snapshot accurately; the snapshots' truth depends on when they were taken relative to other agents' commits.
- **Compounding-confound test fixtures can pass despite a vacuous primary discriminator.** Session 2 R4 (2026-04-26) caught a precedence-surface test that arithmetic-on-paper distinguished pre-fix from post-fix on the precedence key, but two independent confounds masked the bug: (a) `monkeypatch` targeted `dashboard._flag_tags` instead of `watchlist._flag_tags` (the bound reference imported transitively into watchlist module), and (b) ticker names were alphabetically aligned with precedence ordering, so the alphabetical tiebreaker re-ordered the result back into pre-fix-equivalent shape. Generalization beyond the existing `feedback_regression_test_arithmetic.md` discipline: when a test asserts on a primary key, **empirically verify the discriminator by temporarily disabling the keyed-on element and re-running the test** — if it still passes, the test is vacuous despite arithmetic that "should" distinguish. Verified-empirically supplements asserted-arithmetic. Memory file `feedback_regression_test_arithmetic.md` extended with this failure mode in the same housekeeping commit.
- **Production-gating-aware-vs-literal-spec mismatch is a recurring class of finding (3 occurrences as of 2026-04-25).** The pattern: brief specifies a logical rule literally (e.g., "candidates with bucket=watch and only one specific failed criterion"); production's actual gating order makes the literal rule impossible to satisfy on real data because production filters earlier. Three documented cases: (1) candidate-sparsity diagnostic R1 Critical (instrumentation walked emitted criterion order; production gates risk_feasibility first); (2) parity-check R1 (manifest provenance based on commit-time vs run-time state); (3) hypothesis-backend hyp1 R1 (capital-blocked matcher needed `bucket in ('watch', 'skip')` because production's risk_feasibility hard-pre-filter routes capital-blocked candidates to skip not watch). **Generalization:** when a brief specifies a candidate/data filter rule, ALWAYS verify the rule against production's actual gating order (read `swing/evaluation/scoring.bucket_for` and the upstream pipeline). Brief intent (what concept the rule is meant to capture) should be documented separately from the literal rule (which may need to be wider in implementation). Adversarial review reliably catches these; orchestrator should pre-empt by spec'ing intent + verification step in the brief.
- **Sort-neutrality is structurally guaranteed when the new tag never enters the existing sort tuple.** Lesson from chart-pattern flag-v1 brainstorm R1 M2 (2026-04-26). First-cut design tried to modify `_sort_watchlist` to exclude the new flag-pattern tag from `len(tags)`; the cleaner fix was a sibling `pattern_tags` VM field — moves the boundary one layer up and removes the temptation to touch sort code at all. **Generalization for future "decorative tag" features:** keep the sort-input tuple frozen at the existing scoring vocabulary; add new tags as parallel render-only data. If you find yourself touching sort logic to introduce a "neutral" tag, you've already lost — re-architect.
- **ToCToU on form-driven workflows is easy to overlook when "centralizing" looks elegant.** Lesson from chart-pattern flag-v1 brainstorm R3 M1/M2 (2026-04-26). The instinct to centralize cache resolution into `record_entry` was the right "DRY" instinct but ignored the temporal gap between form render and form submit (operator could see one classification, submit, get a different one persisted). Fix: resolve at entry-surface, persist a snapshot + audit anchor; accept hidden-field tampering as residual V1 risk for personal-use scope. **Watch-item template for future form-handling briefs:** "Does the form display a value computed at render time? Does the POST handler re-compute it? If yes, ToCToU."
- **Audit anchors must be PERSISTED to be audit anchors.** Lesson from chart-pattern flag-v1 brainstorm R4 M1 (2026-04-26). R3 added a `pipeline_run_id` field to `EntryRequest` as an "audit anchor" but didn't persist it to the trade row — provenance theater that would have evaporated at record_entry return. **Pattern:** when introducing an audit/provenance field on a transient request type, the next checklist item is "is this column also added to the persistent target row?" If not, it's not really an audit anchor.
- **Schema-layer guarantees beat repo-layer guarantees on every dimension except backward-compat migration cost.** Lesson from chart-pattern flag-v1 brainstorm R2 M2 (2026-04-26). Cache table got a row-level CHECK because it was a fresh CREATE TABLE (cheap); trade table couldn't get one without a heavyweight CREATE-COPY-DROP-RENAME migration, so the constraint went repo-layer-only with explicit residual-risk acknowledgment. **The asymmetry is a real cost** — non-repo writers (e.g., raw SQL via sqlite3 CLI) can violate the trade-row invariant. **Pattern:** when migration cost forces deferring schema-layer enforcement, document the residual gap explicitly and bundle the schema fix with the next column-change migration to amortize the cost.
- **Symmetric refusal across entry surfaces.** Lesson from chart-pattern flag-v1 brainstorm R1 C1 (2026-04-26). The form's "Not classified" stub design for out-of-scope tickers conflicted with the CLI's broad-flag-acceptance design — operator could bypass locked-constraint #5 (cached-only consumption) via CLI. Fix: CLI parity gate refuses `--chart-pattern-operator` without cached row, mirroring form's stub gate. **Pattern:** when one entry surface shows a stub for "feature X is unavailable here," the other surface's analogous control should refuse with an explicit error, not silently allow.
- **Adversarial review verdict ≠ perfection; rounds aren't redundant.** Lesson from chart-pattern flag-v1 brainstorm (2026-04-26). 5-round arc found 22+ distinct findings (2 critical, 11 major, 3 minor accepted-with-rationale, 5+ resolved-by-fix at minor severity). The final R5 minor was a precision nit in a comment that no operator would ever read but a future implementer might. Each round refined a specific failure mode; the rounds aren't redundant work, they're a systematic widening of the spec's defended perimeter. The cycle is worth its cost on high-stakes design work.
- **Synthetic test fixtures must verify parameter→measurement 1:1 mapping empirically before threshold-pair tests are committed.** Lesson from chart-pattern flag-v1 Phase 1 execution (2026-04-26). The original `make_flag_bars` synthetic builder had three latent confounds (pole's first-bar half-band, pre-run-band low, close-relative tightness scaling) that made threshold-pair tests pass-for-the-wrong-reason. Two fix-up commits (`065d368`, `96abfaf`) corrected the mapping. **Generalization:** for any synthetic test fixture for a measurement-gating algorithm, pull representative measurements from the fixture under known parameters and assert the expected values BEFORE writing threshold-pair tests against those parameters. Otherwise threshold-pair tests are vacuous despite arithmetic that "should" distinguish.
- **Detection-outcome tests at search-path-dependent boundaries are NOT gate-correctness tests; pair them with cfg-injection sensitivity tests.** Lesson from chart-pattern flag-v1 Phase 1 (Codex R1 M1, 2026-04-26). The original tightness 0.601/0.599 test happened to pass for the wrong reason — search shifted to a window failing OTHER gates, not because tightness_ratio crossed 0.6. **Fix pattern:** for every gate's threshold-pair test, add a parallel cfg-injection sensitivity test that tightens the gate's cfg threshold and verifies the same fixture flips. The cfg-injection test isolates "is THIS gate enforcing the boundary?" from "does the synthetic happen to detect/reject at these parameters?" Naming convention: `*_above/below_threshold_*` for detection-outcome regression; `*_is_threshold_sensitive` for true gate-correctness via cfg-injection.
- **Reference-enumeration test patterns address determinism + correctness simultaneously for ranking/search algorithms.** Lesson from chart-pattern flag-v1 Phase 1 (Codex R1 M2, 2026-04-26). Tasks 1.11 (search) and 1.12 (best-attempted ranking) originally only asserted determinism (two calls produce same result). After R1 M2, they enumerate all (M, N) candidates via classifier helpers and assert the search picks the EXACT highest-confidence (or max-min-soft-clearance) candidate. Lockstep risk vs the helpers themselves is bounded if per-gate threshold-pair tests catch helper bugs (defense-in-depth). **Pattern:** for ranking/search algorithms, write reference-enumeration tests that compute the expected ranking via the algorithm's helpers and assert the algorithm's verdict matches.
- **Batched-implementation-then-per-element-validation is acceptable for tightly-coupled measurement code IF the plan explicitly documents the chosen pattern.** Lesson from chart-pattern flag-v1 Phase 1 (Codex R1 M4 ACCEPTED, 2026-04-26). Strict per-task TDD red-green-commit is rigid for architectural and stateful code. For tightly-coupled measurement code where the search loop calls every gate, per-gate scaffolding-then-deletion produces throw-away code. Plan deliberately structured Tasks 1.4-1.13 as test-only validation against Task 1.3's batch implementation; plan task text explicitly says "Run to see GREEN (gate already implemented in 1.3)" — adversarial review accepted the deviation because the pattern was documented in advance. **Generalization:** when an algorithm's components are too coupled to red-green-commit in isolation, batch the implementation in one task and use subsequent tasks for per-element test-only validation; document the deviation explicitly in the plan task text so the deviation isn't mistaken for TDD discipline failure.
- **`copowers:executing-plans` → `superpowers:subagent-driven-development` self-collides when multiple subagents pick up the same task.** Lesson from chart-pattern flag-v1 Phase 2 execution (2026-04-26). Phase 2 produced 16 commits instead of the planned 9 because parallel subagents picked up Tasks 2.4 (PipelinePatternClassification dataclass; commits `2d00e9f` + `b8d0435` 82 seconds apart with identical titles + cleanup `5424229`), 2.5 (pattern_classifications repo; commits `22c9f65` + `e3bafba` 42 seconds apart), and 2.7 (insert_trade extension). Implementer self-diagnosed as "an unidentified parallel agent / session" but commit metadata (same author, tight timestamps, identical commit titles) confirmed internal subagent parallelism. ~30% of the chain was cleanup of redundant work. **Mitigation in dispatch briefs (operator decision 2026-04-26):** explicit task partitioning — each task assigned to exactly one subagent; agents may execute multiple tasks; task sets must be DISJOINT across agents; agents must verify task deliverable doesn't already exist before starting (abort + report if it does). Worktree isolation is the heavier fallback if task-partitioning alone is insufficient.
- **Plan-spec test text with literal FK references can render ValueError tests vacuous when PRAGMA foreign_keys=ON.** Lesson from chart-pattern flag-v1 Phase 2 (2026-04-26). Phase 2 plan lines 1620-1670 used literal `pipeline_run_id=1`/`=42` in invariant-test bodies; with PRAGMA foreign_keys=ON (project default), schema-layer FK IntegrityError fires BEFORE the repo-layer ValueError can. Mitigated via `_seed_pipeline_run` helper at execution time. **Watch-item template for future plans:** when literal FK references appear in test bodies, the plan must specify "seed FK target" pattern explicitly (or use sentinel values that don't collide with the FK constraint). Codex review will catch this if the test asserts on `ValueError` and the actual exception is `IntegrityError`, but earlier prevention is cheaper.
- **Cross-column biconditional invariants need exhaustive truth-table enumeration, not incremental invariant listing.** Lesson from chart-pattern flag-v1 Phase 2 (Codex R1 Major 1, 2026-04-26). Phase 2's `_validate_chart_pattern_invariant` was mis-implemented twice: spec-divergence in `c606df1` caught at code review; missing 4th invariant (orphan confidence: `algo=NULL, conf=set, anchor=NULL`) caught by Codex R1 Major 1 fixed in `115c96b`. Listing 3 invariants instead of explicitly enumerating ALL combinations of (algo, conf, anchor) over the {None, "flag", "none"} × {None, set} × {None, set} cube minus the spec's permitted shapes is fragile. **Better template for biconditional invariants:** parameterized table of all combinations → permitted/raises that's exhaustive over the cube; tests parameterize over the same table.
- **Brief artifacts must be committed immediately; untracked briefs are vulnerable to working-tree state issues.** Lesson from chart-pattern flag-v1 Phase 2 dispatch (2026-04-26, refined post-Phase-3). Orchestrator drafted the Phase 2 dispatch brief and left it untracked, expecting the dispatch chain to commit it (as happened with Phase 1's `9f0a778`). Initial diagnosis was that subagent activity deleted the untracked file; **post-Phase-3 verification confirmed the file was actually on disk the whole time** — `ls` returned "No such file" due to transient Windows ACL state on the working directory (the same ACL pattern that blocks scratch-directory cleanup). Actual loss was illusion; recovery was instantaneous when the directory state cleared. **Pattern stands:** untracked files in the working tree are vulnerable to transient inaccessibility (ACL state, subagent activity, IDE indexing); orchestrator MUST commit briefs in the orchestrator-thread immediately after writing them, not rely on the dispatch chain to commit. Robustness is the principle even when the specific failure mode turns out to be illusion.
- **Discriminating-test discipline applies to log-line format assertions, not just test arithmetic.** Lesson from chart-pattern flag-v1 Phase 3 (Codex R1 Major 2, 2026-04-26). Implementer's first attempt at the end-of-step summary log test asserted `"errors" in m`, which would have passed regardless of whether the formula was `total - errors - missing` or `total - errors` or any other variant. Codex caught the substring-match weakness; fix tightened to exact-string assertion `"flag_classifier: 0/1 ok, 1 errors"`. **Generalization:** when the assertion's purpose is to verify EXACT format (log-line schema, JSON-key presence, error-message wording, etc.), substring matching is insufficient. Use exact-equality or structural assertions. Memory file `feedback_regression_test_arithmetic.md` extended with this generalization.
- **Spec/plan formula transcription gap: when the plan provides a code formula, the spec should also specify the formula's semantic intent.** Lesson from chart-pattern flag-v1 Phase 3 (Codex R1 Major 1, 2026-04-26). Spec §3.3 said `flag_classifier: {ok}/{total} ok, {errors} errors` without specifying `ok`/`total` semantics. Plan provided `ok = total - errors - sum(missing)` formula. Implementer faithfully copied the formula. Codex caught the conflation: `total = len(targets)` includes fetcher-failed tickers that never reached the classifier, so the denominator is wrong for "classifier health." **Pattern:** when a plan provides a code formula, the spec should also state the formula's semantic intent in one line so plan-code can be sanity-checked against the spec — not just transcribed from plan to implementation.
- **Subsequent-phase tests can surface earlier-phase contract bugs.** Lesson from chart-pattern flag-v1 Phase 3 (Codex R3 Major 1, 2026-04-26). Phase 3's strengthened happy-path test (asserting full PipelinePatternClassification field-mapping) surfaced that Phase 2's `_row_to_classification` repo function returns ISO date strings instead of `date` objects despite the dataclass being annotated `date | None`. Phase 3 was scope-locked OUT of `swing/data/` (Phase 2 carve-out territory), so the fix was deferred to Phase 4 as Task 4.0a (carve-out extension). **Pattern:** when phase carve-outs lock subsequent phases out of earlier-phase code, the subsequent phase's tests should be the strongest possible regression surface against the earlier-phase contracts; adversarial review at the boundary will catch contract drift; remediation lands in the consuming phase as an explicit prerequisite carve-out extension.
- **Single-subagent dispatch reduces but does not eliminate self-collision.** Lesson from chart-pattern flag-v1 Phase 3 (2026-04-26). Phase 3 was dispatched to a single subagent for sequential Tasks 3.1 → 3.4. Despite no agent-vs-agent collision possible, the subagent produced a rogue duplicate Task 3.2 commit (b080da9) that was immediately reverted (132142c). Most plausible mechanism: subagent context drift causing re-attempt of a completed task without recognizing the prior completion. **Brief discipline + observable verification is the next layer:** subagent must include `git log --grep="Task X.Y" --oneline` output in commit body before each task commit; makes duplicate-task detection observable to Codex review. Worktree isolation is the heavier fallback if the observable-verification layer also fails.
- **Brief task-ID rule scope: task implementations only.** Lesson from chart-pattern flag-v1 Phase 3 (Codex R2 Major 1, 2026-04-26). Phase 2 set the precedent that adversarial review-fix commits don't carry task IDs (commit `115c96b` is named `fix(data): chart_pattern invariant rejects orphan confidence...` without a Task N.M ID). Phase 3 followed the precedent. Codex R2 Major 1 raised the consistency concern; ACCEPTED with rationale (review-fix commits aren't task implementations and don't risk the partitioning failure modes the rule was designed to prevent). **Formalized in standing conventions (Recent decisions 2026-04-26):** task implementations get task IDs; review-fix commits use round + finding ID format; format-only cleanup commits no task ID needed.
- **Single-subagent dispatch + observable verification successfully prevents rogue duplicates** (Phase 4 vindication, 2026-04-26). Phase 4 used the brief discipline + observable verification (subagent must include `git log --grep="Task X.Y"` output in commit body) and produced ZERO rogue task duplicates across 13 commits — Phase 3's `b080da9`-style failure did NOT recur. Single-sequential-subagent dispatch + observable evidence is a working alternative to worktree isolation for the subagent-collision failure mode at this scale. Worktree isolation reserved as fallback for future phases if a new failure mode emerges.
- **Observable-verification grep needs subject-only refinement to eliminate forward-reference false positives.** Lesson from chart-pattern flag-v1 Phase 4 (2026-04-26). The `git log --grep="Task X.Y"` rule generates false positives when commit bodies cross-reference future task IDs in narrative prose (e.g., commit body mentions "Task 4.7 follow-up"). Refinement: `git log --pretty='%s' --grep='^[a-z]+\([a-z]+\): Task X.Y'` — subject-only, anchored to conventional-commit `<type>(<area>): Task X.Y` structure. Eliminates false positives without constraining commit-body prose. Captured in standing conventions; Phase 5+ briefs adopt the refinement.
- **Sort-coupling test vacuousness extension to discriminating-test discipline.** Lesson from chart-pattern flag-v1 Phase 4 (Codex R2 Major 1, 2026-04-26). Even after the R1-split into separate sort-neutrality and wiring tests, the sort-neutrality test was STILL vacuous because the deterministic ticker-ASC tiebreaker made the alphabetically-first ticker (AAA) also the classified one — both correct sort and the hypothetical pattern-tag-first-bug placed AAA first. The fix flipped the ticker assignment so the classified ticker became alphabetically-second; sort-neutrality now requires alphabetical to win OVER any hypothetical pattern-tag bug. **Generalization beyond the existing `feedback_regression_test_arithmetic` discipline:** when designing a discriminating test for an architectural-separation claim, mentally simulate the SPECIFIC bug you're guarding against — is there an output value where the bug AND correct behavior coincidentally agree? If so, the test setup must be INVERTED so the bug's output diverges. Stronger than "verify the discriminator empirically" (which catches re-masking via secondary keys but not symmetry-via-deterministic-tiebreaker). Memory file `feedback_regression_test_arithmetic.md` extended with this generalization.
- **Codex's adversarial review consistently catches subtle test-discipline failures across phases.** Pattern observation (2026-04-26 post-Phase-4 retrospective). Phase 1 caught threshold tests passing for the wrong reason (cfg-injection sensitivity needed); Phase 3 caught log-line substring weakness (exact-equality needed); Phase 4 caught compounding-confound conflation AND ticker-symmetry vacuousness in a 2-round arc. Without the Codex cycle, the tests would have shipped passing-but-not-actually-discriminating. The 5-round adversarial-review investment is yielding repeated ROI on test-quality dimension; not just spec-fidelity.
- **Brief scope-deviation pattern: downstream tests of an authorized carve-out file are naturally in-scope.** Lesson from chart-pattern flag-v1 Phase 4 (R1 Major 2 ACCEPTED, 2026-04-26). Phase 4 brief authorized Task 4.0a Phase 2 carve-out extension touching `swing/data/repos/pattern_classifications.py` (the date deserialization fix). Downstream test file `tests/pipeline/test_step_charts_classification.py` had `.isoformat()` workarounds that became broken once dates became `date` objects — Task 4.8 fixed them. Codex flagged the scope deviation; ACCEPTED as in-scope-by-spirit. **Pattern for future briefs:** when authorizing a carve-out extension touching upstream behavior, the brief should explicitly say "downstream tests of the carve-out file are in-scope by extension" OR explicitly enumerate them. Otherwise legitimate-but-unauthorized work surfaces as scope deviations during review.
- **Base-layout 5-VM rule applies only when `base.html.j2` dereferences the field.** Lesson from chart-pattern flag-v1 Phase 4 (2026-04-26). Phase 4 brief blanket-required all 5 base-layout VMs (`DashboardVM`, `PipelineVM`, `JournalVM`, `WatchlistVM`, `PageErrorVM`) to gain `pattern_tags`. Implementer correctly scoped to 2 (`WatchlistVM`, `DashboardVM`) per spec §3.5 because flag-tag rendering lives in `partials/watchlist_row.html.j2` (consumer-scoped); `base.html.j2` doesn't reference `pattern_tags`. - **Internal code-review BEFORE Codex saves a round.** Lesson from chart-pattern flag-v1 Phase 5 (2026-04-26). Phase 5 plan explicitly anticipated the I1 fix (try/except ValueError around `record_entry`) at lines 3801–3802; implementer missed it during initial implementation; INTERNAL code-review caught it before adversarial Codex round (commit `8507c1a`). Without the internal review pass, this would have surfaced as Codex R1 finding. **Pattern:** running internal code-review pass BEFORE adversarial Codex pre-empts plan-anticipated misses, saves Codex round budget, and tightens the cumulative arc. Implementer sequence: TDD → internal code-review → Codex adversarial review → fix loop. The internal pass costs little; the round saved is meaningful when Codex rounds add 5-15 minutes each.
- **Codex's contextual advantage at finding cross-feature interactions vs internal review.** Lesson from chart-pattern flag-v1 Phase 5 (Codex R1 Major 2, 2026-04-26). The clearest example yet of Codex's whole-diff review catching interactions internal code-review naturally misses. Phase 5 added `chart_pattern_*` form fields; the soft-warn confirm flow (an EXISTING route path Phase 5 didn't directly touch) silently dropped the new hidden inputs on the second-submit round-trip — violating spec §3.6. Internal review naturally focuses on the brief's listed surface area (entry form, POST handler, CLI flag); Codex's whole-diff review caught the soft-warn × new-feature interaction. **Pattern:** when a new feature adds form fields OR persistence fields, ALL existing code paths that re-render the form OR re-submit values are interaction surfaces. Brief should preempt by enumerating them but rarely will catch all. Adversarial review's contextual whole-diff scan is the reliable safety net.
- **Schema-message coupling for IntegrityError mapping is a V1-vs-future-hardening tradeoff.** Lesson from chart-pattern flag-v1 Phase 5 (Codex R2 Minor 1, 2026-04-26). Phase 5's IntegrityError handler in `entry_post` substring-matches SQLite error text to distinguish chart_pattern-related failures from generic ones (treats "FOREIGN KEY constraint failed" as chart-pattern-specific when `cp_anchor_value is not None`). Acceptable for personal-use single-operator scope; documented in `swing/web/routes/trades.py` per follow-up commit `4ef9044`. **Pattern:** when a handler narrows by error-message substring, the coupling is brittle to schema changes; document the coupling explicitly so future schema-change reviewers see the load-bearing assumption. Forward-looking remediation if scope broadens: pre-insert FK existence check OR explicit CHECK-name discriminator.
- **Single-subagent + observable verification VINDICATED 2 consecutive phases.** Lesson refined from chart-pattern flag-v1 Phases 4 + 5 (2026-04-26). Worktree isolation NOT escalated; reserved as fallback for novel failure modes only. **Generalization:** single-subagent dispatch + observable verification + 3-tier commit-message convention is the working baseline for `subagent-driven-development` invocations at this project's scale. Each phase's discipline carries forward to the next; operator decisions about discipline tighten the brief template incrementally.
