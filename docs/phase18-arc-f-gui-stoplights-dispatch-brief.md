## 2. The architecture pass — design (settled, with one decision elevated)

### 2.1 The stoplight framework (provider-driven, forward-compatible)
A `Stoplight` render dataclass — `id` (`tool`/`research`), `label`, `color` ∈ {`green`,`yellow`,`red`,`grey`}, `drilldown_path`. A `health_stoplights(conn, cfg) -> tuple[Stoplight, ...]` aggregator (new `swing/monitoring/stoplights.py`) returns the two stoplights from two **independent providers**:
- **tool-health** — computes at render via 18-E's `compute_tool_health(conn, cfg=cfg, prices_cache_dir=cfg.paths.prices_cache_dir)`; the `ToolHealthStatus.overall` ∈ {green,yellow,red} → the stoplight color. (Cheap DB-backed reads, like the cash-badge precedent.)
- **research** — reads 18-D's JSON status artifact (§2.3); its `overall` → the color; **GREY when the artifact is absent** (18-D not deployed) — the slot is real, the data is pending.
- **`grey` is a RENDER-ONLY state** (no-data / not-deployed / error) — NEVER monitor-emitted (the §3 envelope's `overall` stays green/yellow/red; `ToolHealthCheck`/`ToolHealthStatus` reject `grey` at construction, per 18-E). Provider-driven so a third monitor is a registered provider, not a base-layout re-touch.

### 2.2 THE load-bearing decision — how `health_stoplights` reaches `base.html.j2` (the 500-risk)
`base.html.j2` is shared by EVERY page, so a base-wide field is the documented every-base-VM-or-Jinja-500 gotcha. **Verified on disk (CHARC, not the map): the base fields are hand-duplicated across ~15+ VM classes** — `DashboardVM`, `PipelineVM`, `JournalVM`, `WatchlistVM`, `PageErrorVM`, `ConfigPageVM`, **four `trades.py` VMs**, two `reconcile.py` VMs, three `schwab.py` VMs, `TradeDrilldownVM` — PLUS the `BaseLayoutVM` mixin (`metrics/shared.py:30`, which newer VMs inherit) and the `_base_banner_fields()` factory (`journal.py:235`). Types even diverge (`str | None = None` vs trades.py's `str = ""`). The todo assumed "5 VMs"; it is far worse. So the per-VM-field approach is a ~15-site edit with a real miss-one-and-500 risk.

- **RECOMMENDED — a base-wide injection, NOT a per-VM field.** Inject `health_stoplights` into every template render at the SINGLE template-construction point (`_build_templates` → `Jinja2Templates`, `app.py:86-100`) via a **context processor** (a callable taking `request` → `{"health_stoplights": (...)}`) — or, if that form isn't cleanly supported with the project's `env=` construction, `request.state` set by middleware. `base.html.j2` reads `health_stoplights` from the render context, NOT from `vm`. This **sidesteps the 500-risk and the ~15-VM edit entirely** (there is no per-VM field to forget). **The writing-plans VERIFIES the injection mechanism against the live FastAPI/Starlette version** (the `Jinja2Templates(env=..., context_processors=[...])` signature; or the middleware+`request.state` alternative) — and picks the viable one.
- **FALLBACK (the settled design) — the per-VM field.** If no base-wide injection is cleanly viable, add `health_stoplights: tuple[Stoplight, ...] = ()` to `BaseLayoutVM` + `_base_banner_fields()` + ALL ~15 hand-duplicated VMs, populated by `health_stoplights(conn, cfg)` at each build.
- **EITHER WAY — three binding mitigations (LOCKS §3):** (1) the aggregator/injection is **DEFENSIVE — it NEVER raises** (any failure → grey stoplights), because it runs on EVERY render INCLUDING the error page (a raising base-wide injection 500s every page — strictly worse than the gotcha it replaces); (2) a regression test that EVERY base route renders with the stoplights present + NO 500 (the every-base-route test); (3) the operator browser gate.
- **NOTE (CHARC, deviation from the operator-settled §18-F "per-VM helper" design):** the ~15-VM reality reframes the choice; I recommend the injection. This also does NOT worsen the base-field hand-duplication debt — registered as a debt item (the ~15-VM base-field duplication is a recurring tax; a future deliberate consolidation, NOT bundled into 18-F).

### 2.3 The research-artifact read contract (CHARC-owned; 18-D conforms)
The research stoplight reads 18-D's status from a CHARC-DEFINED path — **`exports/research/health/latest.json`** (a single rolling latest-status file; the `exports/research/` convention per `weekly_glance.py:23`). Define the path as a shared constant in `swing/monitoring/` so BOTH 18-F (reader) and 18-D (writer) reference ONE path (the map confirmed no existing constant). The file's shape IS the §3 monitor-status envelope (`{monitor, generated_ts, overall, checks:[{key,status,summary,detail}]}`). **Absent file → grey** ("research monitor not deployed"). **Malformed/old file → grey + a logged warning** (never a 500; never a false green). **Coordination:** RD's 18-D, when built, writes this path + envelope — flag to RD via the operator (the cross-lane contract, already on record from 18-E).

### 2.4 The drill-down pages (two read-only routes)
`GET /health/tool` + `GET /health/research` — read-only pages (the `journal_trade_detail` route pattern: builder → `TemplateResponse(request, "...", {"vm": vm})`, base fields via the chosen base mechanism), each listing that monitor's `checks` (key / status / summary / detail) so the operator sees WHICH checks flipped. Research drill-down when 18-D absent: a plain "the research monitor (18-D) is not yet deployed" page. The topbar stoplights link to these.

### 2.5 Render + compute
Stoplights render in the `base.html.j2` topbar (`<nav class="topbar">`, `app.py`-built, lines ~68-81) as a `<div class="stoplights">` adjacent to the date — small colored indicators (CSS classes by color) that are `<a>` links to the drill-downs. **Compute-at-render** to start (HTMX poll deferred — todo). Reuse 18-E's `compute_tool_health` (NO new tool-health compute); the research reader + the aggregator live in `swing/monitoring/`.

## 3. LOCKS
1. **NO new schema, NO new dependency.** Compute-at-render (18-E's reader) + a JSON file read; no migration, no pyproject touch.
2. **The base-wide aggregator/injection is DEFENSIVE — it NEVER raises.** Any failure (DB error, a raising `compute_tool_health`, a malformed artifact) degrades to grey, never a 500. It runs on EVERY page incl. the error page; a test forces each provider to raise + asserts grey + no 500.
3. **`grey` is RENDER-ONLY.** Monitors emit only green/yellow/red (18-E's dataclasses reject grey at construction); grey is the framework's no-data/not-deployed/error state. The research stoplight is grey until 18-D's artifact exists, then auto-lights with NO 18-F change (provider-driven).
4. **Reuse, don't fork.** Tool-health = 18-E's `compute_tool_health` (no reimplementation); the research read is the §3 envelope at the shared `swing/monitoring/` path constant.
5. **Browser gate BINDING** (§6) — all-pages base-layout; TestClient asserts body, not the rendered DOM / a base-VM 500 on an unrelated route.
