## 1. The bug (operator-reported, root-caused on disk by CHARC)
Re-authing Schwab via the web (`GET/POST /schwab/setup`) fails with a 500: `PermissionError: [WinError 32] The process cannot access the file because it is being used by another process`, twice consistently. The route's broad except (`swing/web/routes/schwab.py:453`) renders the generic "Unexpected error during setup" 500.

**Root cause (a self-lock):** `swing/web/app.py:191 _construct_web_schwab_client` builds a **LONG-LIVED** web Schwab client (gated on `_is_ladder_active(cfg)` = Schwab production + credentials present), via `construct_authenticated_client` (`app.py:224`). That `schwabdev.Client` holds the per-env tokens DB (`~/swing-data/schwab-tokens.{env}.db`) **open with `BEGIN EXCLUSIVE`** for the whole `swing web` session (the project already documents this lock — see the `del client; gc.collect()` release dance in `auth.py:_assert_v3_tokens_db_loadable_or_raise`, ~`:841-855`). When `/schwab/setup` runs in that same process, the setup flow's **rename-the-stale-DB-aside** step (`os.replace(tokens_db_path, candidate)`, `auth.py:543`, the claim-then-replace) fails — Windows cannot rename/replace a file another open handle holds. The handle is the web app's OWN long-lived client → WinError 32, every time. The setup OSError handling already ANTICIPATES "Windows file-in-use" (`auth.py:344-377`) and degrades gracefully; it does not PREVENT the lock.

**Why now / consistent:** requires Schwab ACTIVE (`_is_ladder_active` true → the web client is constructed). The operator's configured token makes the ladder active, so the web client is live and holds the DB throughout the session — hence every attempt fails.

## 2. The fix — RELEASE-before-replace, then reconstruct
The `/schwab/setup` POST handler must, BEFORE invoking the setup flow that renames/replaces the tokens DB:
1. **Release the long-lived web client** held in `app.state` (Task 0 = confirm the exact attribute/storage on disk — likely `app.state.schwab_client` or similar). Close it + drop the ref + `gc.collect()` so the `BEGIN EXCLUSIVE` SQLite handle is released DETERMINISTICALLY (mirror the existing `del client; gc.collect()` pattern in `auth.py`; schwabdev's connection has no clean `.close()` contract, so the `del`+`gc.collect()` is the release mechanism).
2. Run the setup flow (the rename-aside + the new-token write) — now the rename succeeds (no open handle).
3. **Reconstruct the web client** from the JUST-WRITTEN tokens via the EXISTING `_construct_web_schwab_client(cfg)` factory (NO new construction call site — L2 LOCK preserved), and store it back in `app.state`, so web market-data resumes on the new tokens (else the web falls to yfinance until restart).

**Alternatives weighed (reject):** (a) construct the web client PER-REQUEST instead of long-lived — too invasive (defeats the long-lived-cache perf design + re-opens the v3 construction-at-init hazard per request); (b) retry-with-backoff on the `os.replace` — does NOT fix it (the lock is PERSISTENT, not transient; every retry fails while the client holds the DB).

## 3. LOCKS / disciplines
- **L2 LOCK preserved:** reconstruction reuses `_construct_web_schwab_client` / `construct_authenticated_client` — ZERO new schwabdev client-construction call sites (the `app.py:184` lock).
- **Graceful degradation preserved:** if the release/reconstruct fails, the web degrades to yfinance (the existing `_construct_web_schwab_client` returns-None-on-failure posture) — never a hard crash; the setup itself still succeeds (the token write is independent of the web client).
- **The audit-row + redaction discipline** in the setup flow (`auth.py` `record_call_start`/`_close_audit_error`, the `_redacted_excerpt`) is UNTOUCHED.
- **Concurrent-fetch edge (note, V1):** if a price-fetch is mid-flight on the web client during setup, releasing `app.state` doesn't release that in-flight handle. For the single-operator app this is rare (re-auth is not done mid-fetch). Document it; a stronger V2 could quiesce the price-fetch executor around setup. Not a V1 blocker — flag in the return report.
