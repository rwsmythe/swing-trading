## 2. The verified defect
`swing/web/app.py:302-404` `_install_web_marketdata_caches` builds the web Schwab client (`:314`) and installs two ladder-fetcher **closures that close over that local `client`**: `_quote_hook` (`schwab_client=client`, `:351`, stored via `price_cache.set_ladder_fetcher` `:402`) and `_bars_hook` (`schwab_client=client`, `:379`, stored via `ohlcv_cache.set_ladder_bars_fetcher` `:403`). Both caches live on `app.state` for the app's lifetime. So when the ladder is active the client has THREE strong refs; `_release_long_lived_schwab_client` (in `swing/web/routes/schwab.py`) clears only `app.state.schwab_client` → `gc.collect()` cannot finalize the client → the `BEGIN EXCLUSIVE` tokens-DB handle stays open → `os.replace` still raises `[WinError 32]`. Secondary: even post-reconstruct, the closures would keep referencing the OLD (rotated-token) client.

## 3. The fix (CHARC-specified) — replaceable-holder
Refactor so the two ladder closures **resolve the CURRENT client at call time** rather than capturing the local `client`:
- The natural mutable holder is `app.state.schwab_client` itself (the route already reads/writes it). Plumb `_install_web_marketdata_caches` so the closures resolve the live holder at call time (e.g. pass `app` / `app.state`, or a small shared holder object both the caches and the route reference). **Implementer settles the exact plumbing within this constraint.**
- **None-window degrades to yfinance:** when the resolved current client is `None` (during the release→reconstruct window, or ladder-inactive), the hooks take the existing yfinance fallback path — never crash.
- With this, `_release_long_lived_schwab_client` nulling the holder leaves NO strong ref to the old client → `del` + `gc.collect()` finalizes it → the handle releases → the rename succeeds. `_reconstruct_long_lived_schwab_client` repopulates the holder via the EXISTING `_construct_web_schwab_client` factory → the closures auto-pick-up the new client (also kills the stale-old-client secondary issue).
- Keep the route's release/reconstruct structure from 18-H.4 (it already nulls app.state + gc.collects + reconstructs on all post-release paths); the refactor is what makes that release ACTUALLY finalize the client.
