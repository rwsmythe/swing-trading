## Quick Start

```bash
pip install -e ".[dev,web]"
swing --help                         # CLI: trade, journal, pipeline, web, finviz, etc.
swing web                            # FastAPI + HTMX dashboard on 127.0.0.1:8080
python -m pytest -m "not slow" -q    # fast suite (~1587 tests as of 2026-05-04, ~40-50s)
python -m pytest -m slow             # pipeline/yfinance e2e (minutes, needs network)
ruff check swing/
```

**Windows PATH for `swing` CLI:** `pip install -e` places the `swing.exe` entry point in `%APPDATA%\Python\Python314\Scripts\` (user install, not `C:\Python314\Scripts`). Add that dir to the user PATH permanently via System Properties, or per-session: `$env:PATH = "$env:APPDATA\Python\Python314\Scripts;$env:PATH"`.

**Finviz inbox:** `data/finviz-inbox/` (configured in `swing.config.toml`). Pipeline now fetches the screen via Finviz Elite API automatically (`_step_finviz_fetch`); manual CSV drop remains a fallback override (file-collision skip in the pipeline step). Save manual exports as `finvizDDMmmYYYY.csv` (1-2 digit day, 3-letter month, 4-digit year — e.g. `finviz5May2026.csv`). API-emitted CSVs use the same filename pattern, anchored on `action_session_for_run(datetime.now())` per [`swing/pipeline/runner.py:_step_finviz_fetch`](swing/pipeline/runner.py). Validator requires 13 columns: `No., Ticker, Sector, Industry, Country, Price, Change, Average Volume, Relative Volume, Average True Range, 52-Week High, 52-Week Low, Market Cap`. Missing columns → CSV moved to `data/finviz-inbox/rejected/` with a sidecar `.rejected-reasons.json`. Pipeline outputs go to `exports/<action_session_date>/` (briefing.md + briefing.html + charts/).

**Pipeline lease-acquisition wait** (`web.pipeline_lease_wait_seconds`, default 5s): how long POST `/pipeline/run` waits for the spawned subprocess to insert its `pipeline_runs` row. Python 3.14 cold-start + heavy imports on Windows regularly cross 2s; bump if you see false "did not acquire lease" errors on a slow box.

- `data/` — SQLite schema (`migrations/*.sql`), dataclass models, repo functions

## Gotchas

- **yfinance rate-limits.** Use `threads=False` on `yf.download()` ONLY. `yf.Ticker(t).history()` does NOT accept `threads=` and raises `TypeError: got an unexpected keyword argument 'threads'` on yfinance >= 1.2. Concurrency for `Ticker.history()` is bounded by the app-level executor; `PriceCache` + `OhlcvCache` sliding-window breakers are the real backpressure.
- **Test-count drift in plan docs.** Plans show stale expected counts — trust `pytest` output.
- **The auto-memory at `~/.claude/projects/c--.../memory/`** can go stale. Verify `project_refactor_intent.md` against current git log before assuming refactor is out-of-scope.
- **HTMX 4xx fragments need an explicit config override.** HTMX 2.x default is `{code:"[45]..",swap:false,error:true}` — 4xx responses fire the error event but don't swap. `base.html.j2` contains a `htmx.config.responseHandling` override that enables 4xx swapping; preserve it if you touch the base layout. TestClient-based tests won't catch a regression (they assert response body, not DOM state).
- **Starlette middleware is LIFO.** `add_middleware` call order is the REVERSE of request execution order — later calls become more-outer. Security-critical middleware (OriginGuard strict-mode, request-id stamping) must be added LAST so it wraps everything inside. A body-size check added after OriginGuard but before RequestId would fire 413 before OriginGuard's 403, silently weakening strict mode.
- **`os.replace` requires same filesystem.** On Windows with Drive-synced paths + `$TMP` on a different volume, `os.replace(tmp, final)` raises `OSError: [Errno 18] Invalid cross-device link`. For atomic-replace flows: create temp files in the destination directory (`tempfile.NamedTemporaryFile(dir=<dest_dir>, ...)`). Never use `shutil.move` expecting overwrite on Windows — it fails when the destination exists.
- **yfinance `history(interval="1d")` includes the in-progress bar during market hours.** The last row is today's PARTIAL session until the next session rolls over. If you compute a rolling-mean SMA or a "previous close" directly off `df.tail()`, you're sampling a partial intraday close — a "close below MA" rule turns back into an intraday rule. Strip it: `if df.index[-1].date() >= session: df = df.iloc[:-1]` where `session = action_session_for_run(datetime.now())`. Use the exchange-session helper, NOT `date.today()` — HST lags ET by 5h, so local midnight would incorrectly preserve today's partial bar for hours.
- **`base.html.j2` is shared — new `vm.foo` field requires adding to EVERY base-layout VM.** Every page that `{% extends "base.html.j2" %}` has a VM rendered as `vm` in the template context. If the base layout starts dereferencing a new field (banner flag, session date, etc.), EVERY base-layout VM must gain that field (with a safe default) or Jinja 500s unrelated routes with `UndefinedError`. Current set: `DashboardVM`, `PipelineVM`, `JournalVM`, `WatchlistVM`, `PageErrorVM`. Phase 3c hit this with `price_source_degraded` initially and caught it during adversarial review; Phase 3d hits it again with `ohlcv_source_degraded`. Same fix pattern each time.
- **Cache + executor race: workers must not write to shared state when the request thread cancels on deadline miss.** `Future.cancel()` is a no-op on a running worker. If the worker writes to `cache._store` from inside itself, a late-completing worker will poison the cache with a bundle the request thread already gave up on. Pattern: worker returns a pure value; request thread writes to `_store` ONLY for futures that completed in-deadline (`wait(..., timeout=deadline)` → iterate `done`, not `pending`). `OhlcvCache` follows this; future caches should too.
- **Weather lookup in read-only UIs must NOT query by `action_session`.** Weather is stored keyed by `data_asof_date` (last completed NYSE session); `action_session` is forward-looking (next session). On Sunday-evening → Monday-prep workflow (or any weekend/holiday gap) the query silently returns `None` → UI renders STALE despite a fresh weather row existing. Use `swing.data.repos.weather.get_latest(conn, ticker=...)` which returns the most recent record by `run_ts` regardless of date. The pipeline's own briefing generator uses `data_asof`; read-only UIs should use `get_latest`.
- **OHLCV fetch scope = open-trade tickers ONLY.** Advisories (trail-MA, exit-below-MA) only fire on open positions; watchlist rows never consume SMA data. Fetching OHLCV for watchlist tickers burns yfinance quota on every render and trips the `OhlcvCache` sliding-window breaker under normal load. `build_dashboard` computes `ohlcv_tickers = sorted({t.ticker for t in open_trades})` — do NOT union with watchlist or `active_tickers`. With zero open trades, skip the fetch entirely (guarded by `if ohlcv_cache is not None and ohlcv_tickers`).
- **yfinance `group_by='column'` now returns a MultiIndex column** (`Price × Ticker`) even for single-ticker calls. `df["Close"]` is a DataFrame, not a Series — `float(iloc[-1])` raises `TypeError: float() argument must be a string or a real number, not 'Series'`. Squeeze defensively: `close = df["Close"]; if hasattr(close, "ndim") and close.ndim == 2: close = close.iloc[:, 0]`. This and the `Ticker.history()` `threads` regression are examples of the same failure mode: pip-installed dependency updates silently break integration points that were working last quarter.
- **`exchange_calendars.is_open_at_time` requires `pd.Timestamp`**, not `datetime.datetime`. Passing a plain datetime raises `TypeError: \`timestamp\` expected to receive type pd.Timestamp`. The prior `market_hours_now()` caught all exceptions silently and returned `False`, permanently routing PriceCache to the last-close fallback during live trading hours. Wrap: `pd.Timestamp(datetime.now(timezone.utc))`. Same failure class as the yfinance API regressions above — when you catch broad exceptions around a dependency call, log them instead of silently returning a degraded default.
- **`PriceCache._last_close` only sees tickers in today's `candidates` table.** Open-trade tickers that rotate OUT of the finviz CSV get no fresh candidate row, so the after-hours fallback returns the close from whenever they were last in finviz (potentially days old). `_step_evaluate` unions `list_open_trades` into its fetch loop and writes them as `bucket='excluded'` with `notes='open position'` so the close stays fresh every run. If you add a new "priced" UI surface, verify its tickers actually appear in `candidates` — otherwise the fallback lies silently.
- **Queries ordered by `started_ts DESC` on `pipeline_runs` mask prior completes mid-run.** A new run with `finished_ts IS NULL` wins the ORDER BY and surfaces NULL where callers expect the last-known-good timestamp (dashboard "Last pipeline" rendered "never" the entire time a run was in flight). Two-read pattern: most-recent COMPLETED row's `finished_ts` for "when did we last have good data?", separately most-recent-started row's `state` for "what's happening right now?"
- **HTMX OOB-swap partials that hand-duplicate full-page markup drift silently.** The refresh-now `prices_refresh_container.html.j2` partial hand-duplicated the watchlist `<table>` markup without the `<h2>` heading and "Show all (N)" link that live in `watchlist_top5_section.html.j2`. On OOB swap the heading vanished and the watchlist butted up against the Open Positions table; a hard browser refresh restored the layout because full-page render used the canonical partial. Cosmetic symptom, structural cause. Fix: OOB-swap partials and the full-page render must go through the SAME `{% include %}` target, not hand-duplicated markup. If you find yourself copying JSX/HTML fragments between two partials to avoid an include, stop — the drift failure is a matter of when, not if.
- **External-API empty-result must be treated as transient when write-through-caching.** Failure mode discovered 2026-04-30 (OHLCV archive helper R2 Major 2; commit `3335d6c`). Write-through caches/archives backed by external APIs (yfinance, broker APIs, weather APIs) must NEVER replace existing cached content from an empty API response. yfinance occasionally returns empty for transient reasons — rate limit, network blip, weekend/holiday, ticker-symbol handling drift, upstream regression. Without the fallback, an established ticker's archive blanks; subsequent reads must re-fetch full history; historical context is permanently lost if the next call also returns empty. **Append-or-fall-back pattern:** on empty API response, RETAIN existing cached content; leave stale metadata (so next call retries the refresh); never blank. The only case where empty-as-authoritative is correct is when the API has explicit "no data found" semantics distinct from "transient failure" — and even then, prefer leaving stale data + flagging staleness over blanking. Applies to: `swing/data/ohlcv_archive.py` (now correctly handles); future weather caches; future broker-API caches; any write-through-cache backed by a flaky external API.
- **HTMX response leading with `<tr>` triggers `makeFragment` synthetic-table-wrap that drops `<table>` content inside OOB `<section>` chunks.** Failure mode discovered 2026-04-29 (entry_post Bug B; commits `398483d` + later). When an HTMX response's primary content starts with a `<tr>` element, the htmx.js `makeFragment` helper wraps the entire response in a synthetic `<table><tbody>` so the browser's HTML5 parser will accept the `<tr>` as valid (table-row content can't be parsed at fragment root). HTML5 nested-table parsing rules then DROP any inner `<table>` elements found inside `<section>` OOB chunks during the fragment parse — `htmx:oobBeforeSwap` and `htmx:oobAfterSwap` fire successfully, but the swap content has been MANGLED at parse time, leaving the OOB target with only the wrapping non-table elements (e.g., `<section><h2>Watchlist</h2></section>` with the `<table>` of rows silently gone). Server-side TestClient assertions on response body show fully-populated content; the bug only surfaces in a real browser's parser. **Fix: pure-OOB response architecture** — don't lead the response with a `<tr>` at fragment root; deliver new rows via OOB swap into the destination tbody instead (mirroring `partials/prices_refresh_container.html.j2`'s pattern). **Pre-empt the failure mode in any new HTMX endpoint that returns multiple swap chunks: keep the response's primary content table-row-free (use `<div>` or `<section>` at the root if you need primary content at all; otherwise return only OOB chunks).** This is the canonical example of the JS-test-harness gap (orchestrator-context "Lessons captured" 2026-04-29) — TestClient response-body assertions cannot catch this; only operator-witnessed browser DevTools verification (or a Playwright/Selenium harness) will.
- **Matplotlib mathtext fires on `$` (paired math mode), `^` (superscript), `_` (subscript), and unbalanced `\` — silently italicizes intervening text, consumes the metacharacters, and collapses surrounding whitespace.** String-equality test assertions verify the format STRING; they do NOT verify the RENDERED output through the mathtext interpreter. Commit `2fd0ecc` attempted to escape `$` with `\$` in the chart title format; matplotlib resolves `\$` to a literal `$` BEFORE math-mode parsing, so the paired `$..$` still triggered math mode (italics on "stop", `$` glyphs consumed, spaces collapsed in the rendered PNG) despite the test strings reading `\$110.00`. Real fix (`29c93f5`): omit the metacharacter from the title format entirely. Pattern: when adding/changing any rendered text that flows through matplotlib (`title=`, `xlabel=`, `ylabel=`, axes annotations, suptitles, legends), or any mathtext-aware renderer, **manual visual verification of rendered output is non-optional** — string-equality on the source string is insufficient. Future title-format additions (scientific notation `1e3`, exponents `x^2`, subscripts `x_i`, footnote markers) must visually re-verify on rendered PNG. Defense-in-depth alternatives if the threat model widens: `fig.suptitle(..., parse_math=False)` after `mpf.plot(returnfig=True)`, or a `_sanitize_for_mathtext()` helper that escapes/strips metacharacters at format-string time.
- **HTMX form-driven endpoints have two browser-only failure surfaces TestClient cannot detect: HX-Request header propagation on embedded forms + HX-Redirect for success-path response.** Failure modes discovered 2026-05-02 (Phase 5 config page, Codex R1 Major 1 + 2; commit `4d3174d`). (1) Embedded forms inside HTMX-rendered fragments must include `hx-headers='{"HX-Request": "true"}'` on the form element — without it, OriginGuard strict-mode rejects subsequent submits with 403 because the parent page's HTMX header context doesn't propagate to nested forms by default. TestClient passes because tests send the header explicitly; real browser submissions without the propagation hint fail. (2) Success-path response to HTMX POST should be `204` + `HX-Redirect: <url>` header (browser re-navigates via htmx.js), NOT a `303` → swap-target — real browsers using htmx.js swallow `303` transparently and the swap target receives nothing. TestClient verifies the `303` status code but not how htmx.js processes it. **Pre-empt in any new HTMX-driven route handler with form submission:** the plan's per-task acceptance criteria MUST enumerate (a) embedded-form `hx-headers='{"HX-Request": "true"}'` propagation under OriginGuard strict-mode, and (b) success-path `HX-Redirect` emission instead of 303 swap-target. Operator-witnessed browser verification gate is BINDING for HTMX work — both failure modes are invisible to TestClient. Pattern complement to the JS-test-harness gap lesson family (Bug 1 follow-ups 2026-04-25; hyp-recs success-path 2026-04-29; entry_post Bug B `<tr>`-leading makeFragment 2026-04-29).
- **HX-Redirect target route must be verified to exist; TestClient verifies the header but doesn't follow.** Failure mode discovered 2026-05-04 (Phase 6 review POST, operator-witnessed verification gate; fixed inline as code-review I3 on worktree). - **Python `... or ""` idiom collides with SQL CHECK-constraint nullability when CHECK enum doesn't include empty string.** Failure mode discovered 2026-05-04 (Phase 6 `mistake_cost_confidence` form input; deviation #3 from executing-plans return report; corrected in implementer-side fix). Plan template had `value = form_value or ""` for nullable text column with CHECK constraint `mistake_cost_confidence IN ('high', 'medium', 'low')`. The `or ""` idiom produces an INSERT that fails the CHECK (empty string isn't in the enum) instead of writing NULL (which a nullable column allows). Fix: `value = form_value or None` for nullable text columns with CHECK constraints. Subtle Python-idiom-vs-SQL-nullability collision; recurs anywhere a form-input fallback feeds a nullable enum-CHECK column. Pre-empt in any new schema-with-CHECK + form-input combo: prefer `... or None` over `... or ""` for nullable text columns; add a discriminating test that submits an empty form field + asserts NULL persisted (not empty-string CHECK violation).
- **SQLite `INSERT OR REPLACE` (and the bare `REPLACE` shorthand) is `DELETE old + INSERT new` semantically — NOT a true UPDATE.** Failure mode anticipated 2026-05-06 (Phase 8 daily-management brainstorm Codex R4 Major fix; spec at `docs/superpowers/specs/2026-05-06-phase8-daily-management-design.md` §4.2). For any audit-trail-with-uniqueness pattern, REPLACE will: (a) CASCADE-WIPE child rows referenced via ON DELETE CASCADE FKs, (b) issue a NEW auto-increment PK (history-rewrite), (c) reconstitute index entries with the new PK. Symptom on Phase 8's daily_management_records: same-tier reflow via `INSERT OR REPLACE` would silently destroy any audit-FK chain pointing at the prior row's PK + cascade-wipe linked event_log entries that reference it. **Pattern: SELECT-then-UPDATE-or-INSERT** — query against the unique-key tuple to find existing row, UPDATE if found OR INSERT if not. Same family as the `executescript()` partial-failure rollback gotcha — both about SQLite-runner-level discipline that needs explicit code, not implicit assumptions about ORM-style atomicity. **Pre-empt in any new UPSERT design:** writing-plans phase enumerates SELECT-then-UPDATE-or-INSERT semantics; reject `INSERT OR REPLACE` proposals on tables with FK references or audit-trail intent.

- **Service-layer `with conn:` opens its own transaction — DO NOT call from inside an outer single-transaction; call the REPO-level function instead.** Failure mode anticipated 2026-05-06 (Phase 8 daily-management plan §A.1; binding contract in `swing/trades/daily_management.py:record_event_log`). Test rollback semantics through the actual runner path, NOT bare `executescript()+rollback`; a bare `executescript()` test will assert `probe_table absent after rollback` and fail because the implicit COMMIT already persisted the row. Discriminating test pattern: malformed migration with deliberate fail-mid-sequence; assert post-failure that the runner rolled back AND `conn.in_transaction == False`. **Forward-relevance:** Phase 9/10 schema migrations + Schwab API integration if it adds tables + any future migration-runner work inherits this discipline.
- **`in_transaction` "auto-detect outer transaction" guards in transactional services re-introduce the very race the explicit lock was meant to close — prefer "reject + simple contract" over "auto-detect + complicate".** Lesson from Phase 8 `record_event_log` Codex R3→R4 cascade (2026-05-07). Initial implementation used Python's `with conn:` (deferred transaction); R3 added explicit `BEGIN IMMEDIATE` to close the stale-prior_stop race; R4 caught that the `in_transaction` guard added in R3 actually re-introduced the race because the guard's "auto-detect outer transaction" logic could deceive a caller into thinking they were inside the lock when the connection's deferred transaction had not yet escalated. Final contract for any single-transaction service: **caller MUST NOT hold an open transaction; function ALWAYS owns BEGIN IMMEDIATE / COMMIT / ROLLBACK; reject (don't auto-detect) caller-held transactions.** Pre-empt in any new transactional service via writing-plans §5: rejection-on-caller-tx contract explicit + discriminating test asserts function raises if called inside open transaction.
